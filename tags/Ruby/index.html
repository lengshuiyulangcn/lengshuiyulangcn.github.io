rack 中间件 warden 的机制

5月 31, 2016

说到warden，大概很多人感觉很陌生。然而很多人耳熟能详的Devise就恰恰是基于warden构建的。
所以当我们customize Devise的一些功能的时候常常看到有warden这个关键词的出现。
要深入了解Devise的验证机制，不如先从了解warden这个东西入手。

概述

warden在rack中间件的堆放位置大抵位于session下流。

其中，default_strategies 是warden默认使用的验证策略。而failure_app则是当验证失败的时候处理相关内容的中间件。

warden 通过strategy来定义用户验证策略。

authenticate!则是真正验证的方法。比如上面的例子里面我们看到其调用了User的authenticate方法对用户名密码进行验证。

在验证的时候我们可以用到的一些方法有

  • request – Rack::Request object
  • session – The session object for the request
  • params – The parameters of the request
  • env – The Rack env object

比如上面我们用params获取了传来的用户名和密码参数。

进行验证之后，根据验证结果，我们可以有一下处理。

  • halt! – 停止接下来所有的strategy。当前的strategy会作为最后一个策略。
  • pass – 直接忽略次策略。当然不推荐就是了。ignore this strategy. It is not required to call this and this is only sugar.
  • success! – 策略成功。因为warden的策略有一个成功即会停止，所以紧接着会造成halt!也发动。
  • fail! – 策略失败。并发动halt！
  • redirect! – 跳转到另一个url。同样发动halt!
  • custom! – 返回一个自定义的rack array并造成halt!

感悟

当Devise太重的时候我们完全可以通过定制自己的warden strategy来定制我们自己的用户验证机制。

Rack的层层堆砌逐渐由模糊到清晰起来。

Ruby-china 的技术备忘

Ruby-china的代码感觉可读性很高，并且更新较为频繁，也用到了一些自己不熟悉的技术，首先还是先来整理一下这些平时不太接触的技术吧。

Elasticsearch

强大的站内搜索工具。配合ruby的wrapper
elasticsearch-rails可以实现强大的全文搜索功能。感觉大抵和solr＋sunspot类似。另外对于中文搜索需要配合一些tokenizer。

Postgresql

虽然是关系型数据库，但是支持json。RubyChina之前使用mongodb，最近切换到了postgresql。

Actioncable

Rails 5的新特征。其实就是websocket的Rails实现。

second_level_cache

很酷炫的一个gem。second_level_cache直接override了ActiveRecord的find方法，find方法的结果会直接被cache，但是不用担心，当发生update，delete，create等变化的时候，cache的内容也会相应得到变化。

需要注意的是，这个gem的作者推荐大家使用dalli_store作为默认的store，这大概也是Ruby-china使用dalli_store的原因了。

rack-attack

纯引用。Rack::Attack is a rack middleware to protect your web app from bad clients. It allows whitelisting, blacklisting, throttling, and tracking based on arbitrary properties of the request.

一些感想

阅读ruby-china的源码让无聊枯燥的周末下午突然变得很有意思。
作为一个ruby开发者，再次深刻体会到ruby以及rails世界的深奥和广大。
当然ruby-china也有一些地方美中不足。
比如对cache依赖较多，而没有使用一些前段技术来提升用户体验和速度。
比如很多地方没有针对开发者的优化。
文档还不是十分完善，部署较为复杂，容易使得初学者望而生怯。

eager_load 和 autoload

eager_load顾名思义就是饥渴地load，而autoload看不出来什么名堂。
一般来说，autoload的模块在Rails中只有被调用的时候才会加载。而eager_load的模块则是从一开始就被加载了。
Rails中，development mode启动较快，那是因为对很多模块Rails启用了autoload。
而在production mode中，Rails则对这些模块启用eager_load。

这说明，Rails对一些默认的加载的模块(ex./app/*)按照不同的环境采取了不同的加载策略。而我们自定义的模块，该如何将其也和Rails的策略同步，使得在development mode下 auto_load， 在production mode下eager_load呢？

eager_autoload

讲了那么多eager_load 和 auto_load, 我们终于可以来讲讲eager_autoload
参照最上方的例子。我们在module中Hoge中想要调用Foo和Bar的时候，我们也得考虑如何加载Foo和Bar。
而使用eager_autoload的最大好处就是，当我们eagaer_load Hoge的时候，Foo和bar也被eager_load。当我们用auto_load加载Hoge的时候，Foo和bar也同样是auto_load。

又是一年感恩节。昨晚久违玩游戏玩得太晚，爬起来才发现已经是午后。随便超市里买点垃圾食品吃了不知不觉又玩了一下午游戏…

晃到星巴克才坐下，昨天debug了半天修复的问题，今天又有了相关的新问题。

还是从昨天的问题反省开始。

昨天的问题是monkey patch devise的method的地方忽略了strong parameter。

修复方法是给conditions套上strong parameter。

邮件正常发送，但是，用户收到邮件点击重置密码的url，输入新密码确认提交之后，居然发生了conditions没有permit方法的错误。
检查一下conditions是啥玩意儿。

昨天的那个啥呢？

原来ActiveSupport::HashWithIndifferentAccess是 ActionController::Parameters的父类。

而那些strong parameter的玩意儿，则是ActionController::Parameters特有的。

可是既然ActiveSupport::HashWithIndifferentAccess不支持permit，那么我们该肿么办？

to_h and to_hash

很有意思看到了这个ticket。https://bugs.ruby-lang.org/issues/10180
挺有意思的是，to_h是Hash自带的方法，而然，通常当我们需要非显示得将object转换成hash的时候，
通常可以试一下to_hash这个方法。to_hash本身需要在后继的class中定义。够lucky的是，ActiveSupport::HashWithIndifferentAccess中定义了to_hash这个方法。

于是昨天和今天的问题综合起来可以如下解决。

validation sucks

其实调查这个问题之前我想到的是另一个问题，即用户修改密码的时候会出发validation， 然后因为仅仅修改密码而没提供其它filed的数值导致user的update失败。这个问题一直困扰了我很久。

但其实仔细一想，我们无非就需要override devise的修改用户密码的method，在重载的method中忽视validation。而不敢深入devise的源码则是硬伤。

以上就跳过了修改密码时候的user validation。

Sunny Bug Day

2月 13, 2016

之前有一阵子收到了很多重置admin密码的邮件。一直当然是有人恶作剧，直接无视了。
直到今天才确认用户发送的所有重置password邮件由于某种原因全部发送给了admin。于是开始着手调查。

enter point

找到最初发送重置密码邮件的地方。

password_path(resource_name) 对应的controller和action则是devise/passwords_controller#create

在这个方法中设置一下断点看一下resource_params

貌似ok。问题应该是在resouce_class.send_reset_password_instructions()这个method里面了。

resource_class 在这个例子中则是User这个Class。send_reset_password_instructions由devise通过active concern插入到User类中。

这个方法又调用了find_or_initialize_with_errors。然后这个find_or_initialize_with_errors会接着调用

问题的切入点大概已经浮现了，就是这里了。find_first_by_auth_conditions，由于之前我们在Rails的User model中monkey patch了这个devise的方法。所以导致我们并没有使用devise的原生态的这个方法。

之所以要在application中重载这个方法是因为Qilian需要同时兼容用户ID和邮箱登陆。也就是说在登陆框中输入ID或者Email，都可以登陆。
这里的问题不是很显而易见，因为大多数情况下我们我们都没有走到if 条件的else中。然而这次，由于不是login，我们来到了else。查看一下conditions。

原因已经是明白了。原本我们要通过conditions查找有该邮件的用户。然而由于conditions.to_h返回了一个空hash，导致where检索返回全部用户，于是我们每次取的都是这次检索的第一个用户，在现在的情况下正好是admin。
conditions到底是个什么鬼。

看到这里大概应该恍然大悟了。其实之所以to_h返回了空hash是因为Rails4.0 以后引进了strong parameter。而ActionController::Parameters
是继承自hash的类，不过其to_h被后续的模块给hook然后添加了过滤验证了。

How to Fix

按照惯例给condtions加上了strong parameter的验证。 一切OK。

Things learned

这半个下午的折腾可以说是很有价值的。因为这让我们重新认识到了一些在开发过程中很重要的事情。

  1. monkey patch 真的很难debug

  2. 在gem中通过raise调用better error debug请注意重启。
    这个是在浪费了很多时间时候才突然想到的。平时在rails的dev mode我们习惯了用raise来调用better error的网页debug。之所以我们可以直接插入raise，直接刷新网页就可以debug这个还是多亏了rails的spring这个东西。这个东西就像grunt的watch一下默默的帮我们做了很多事情。而然，当你要在gem中使用raise，千万别忘了重启服务器，因为gem中的一些方法之类都是在启动过程中插入到rails中的。

  3. copy and paste 之前还是得三思。这次是因为图省事超的一段代码年代过久，在strong parameter之前的时代虽然是ok的，但是如今已经ng。

离成为一个出色的Ruby开发者看来还有很长一段路要走。
