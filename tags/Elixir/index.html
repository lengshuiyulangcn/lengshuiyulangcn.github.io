Macro.escape

懵了。为什么a,b,string这些东西不需要Macro.escape, 而p需要呢？

按照文档的说法，

it is important to make a distinction between a regular Elixir value (like a list, a map, a process, a reference, etc) and a quoted expression. Some values, such as integers, atoms and strings, have a quoted expression equal to the value itself. Other values, like maps, need to be explicitly converted.

这是神马意思？

就是这个意思。 1 的 quote expression 还是 1， 而 %{name: "www"}的quote expression则是 {:%{}, [], [name: "www"]}
Macro.escape的作用就是把value转换成quote expression。

由于定义macro的时候，在quote...do中我们需要用unquote去评价 quote expression。所以，先得把map这种类型转换成quote expression。

俗而不凡的Plug

10月 30, 2016

Plug这个名字作为Elixir Web框架Phoenix 中间件的名称，俗得可以。
但是Plug的思想十分单纯，基本就是为了实现和Rack相同的中间件功能。
然后和Rails大不相同的是，Rails中router以后的部分我们基本无法定制中间件。而Plug可以轻轻松松插到controller中间。controller的action本质上也是Plug，所以在Phoenix 框架中，从受理request到回复response，这一系列中间过程其实就是串联起来的Plug来完成的。

Module Plug 和 function Plug

要写一个Plug很简单。首先以function plug为例子。

任何第一个参数为Plug.connect struct，并返回Plug.connect struct的带两个参数的函数都可以function plug。
比如上例，我们向Plug.connect写入了一些http的header，最后返回一个带200 status，body为”Hello world”的 Plug.connect struct。
其实作为function的Plug，我们并不关心函数的第二个参数是什么。

那么第二个参数是用来干什么的呢？ 那自然是Module的Plug中会用到。
所谓Module的Plug，需要实现两个函数。第一个是init/1函数，而第二个是call/2函数。

再来看看Phoenix框架中使用Plug.Session的例子。

可以看到init/1函数的参数来自plug plug_name, opts 的opts。

Session in Cookie 的实装 Plug.Session.COOKIE

Plug.Session 是如何开放接口提供调用Plug.Session.COOKIE中的方法呢。
首先Plug.Session 和Plug.Session.COOKIE虽然看起来前缀相同，但是Elixir中，module的是扁平的，也就是说，
这两个module没有一毛钱关系。

注意看 Plug.Session 的 init/1函数，store = Keyword.fetch!(opts, :store) |> convert_store
按照 lib/hello_phoenix/endpoint.ex 中的配置，Keyword.fetch!(opts, :store)的结果应该是:cookie。
那么store的结果就应该是:cookie|> convert_store的结果了。

终于是激动人心的convert_store/1了。

卧槽 "Elixir." <> _ -> store到底是什么？
再来复习一下Elixir的pattern match吧。

在这个例子中，Elixir 会把 {1,3,4} 和 {1,x,4}进行 match。 这个时候，x匹配上了4。
再来看"Elixir." <> _ -> store， 其实就是，如果 store以 Elixir.开头，那么我们返回store。
当store为:cookie时候，条件落到第二个，reference 和 :cookie匹配， case ...do的返回结果则是将 Plug.Session 结合到 "cookie"前面，最后结果为":Plug.Session.COOKIE"。

哦，原来如此，这样使用store.put/3其实就是在调用Plug.Session.COOKIE.put/3了。
这样的话，我们就可以自由扩展Plug.Session来写我们自己的Session存贮了。
其实，Plug为Session的扩展提供了Protocol。

Thoughts

据说Plug的函数实现每次需要带一个不用的参数其实也是神烦。不过不实际写一下还是没什么实感。
