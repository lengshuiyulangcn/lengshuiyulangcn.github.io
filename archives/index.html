没有干劲的周末以及 Electron

10月 02, 2016

周末抽时间看了一下时下很火的Electron。
Electron是基于google V8 以及 chromium 的跨平台CS桌面应用框架。
通过 NodeJS衔接 OS 的I/O 操作， 通过 CSS/JS/HTML 来渲染UI。

和大部分桌面应用的区别就是，Electron 通过 Node 进行 OS 的I/O 操作，而传统桌面应用使用OS提供的API。
Electron 通过 CSS/JS/HTML 来渲染UI， 而传统桌面应用使用OS提供的API。

大名鼎鼎的编辑器Atom就是基于Electron的跨平台应用。
其他比较有名的应用，比如Slack也是基于Electron。

quick-start

Electron的核心可以是一个main.js

main.js 作为主进程，需要在其中创建一个窗口，将需要载入的页面载入该窗口中，绑定一系列监听时间来对用户的操作作出反应。

由于内容一目了然，具体我就不抄了。

能干什么

基本的本地应用程序基本都能用Electron实现。现在间的比较多的有Markdown编辑器，文法高亮编辑器等等。
Markdown编辑器的思路很简单就是和网页上用Marked.js实现的完全一样，再附加保存到本地功能和从本地文件系统读取而已。
文法高亮编辑器配合JS的编辑器ACE，也就是多个本地读写的功能而已，这些都不用多说。

与http restful API交互可以使用superagent之类的http客户端，实现复杂的UI也可以套用React和Redux以及其他一大波前端框架。

总之不管如何，Electron的本质其实就是获得了本地IO操作能力的Web APP。

没有干劲的周末

大抵工作上不爽是最大的原因。

  • 十分讨厌PHP以及写得很屎的PHP。

  • 觉得周围很多人很蠢或者在混日子。

  • 工作的环境不适合自己的技术成长。

  • 觉得用户是傻逼。

  • 觉得工资太低。

  • 觉得把妹太难。

  • 我也是傻逼。

还是趁机汲取为数不多的能学到的东西，找好下家吧。

要注意的是，warden并不会主动来验证并拒绝request。只有当我们调用以上的验证方法的时候，warden才会横空挡路。
这也就是为什么我们使用devise的时候，通常在controller的before_action hook下面写上authorize_user!的原因了。

strategy策略

warden 通过strategy来定义用户验证策略。
我们来看一下password strategy的实现。

valid?方法作为strategy的哨兵，在验证策略发动前进行一些检查。如果不写这个方法，那么strategy总是会发送。

authenticate!则是真正验证的方法。比如上面的例子里面我们看到其调用了User的authenticate方法对用户名密码进行验证。

在验证的时候我们可以用到的一些方法有

  • request – Rack::Request object
  • session – The session object for the request
  • params – The parameters of the request
  • env – The Rack env object

比如上面我们用params获取了传来的用户名和密码参数。

进行验证之后，根据验证结果，我们可以有一下处理。

  • halt! – 停止接下来所有的strategy。当前的strategy会作为最后一个策略。
  • pass – 直接忽略次策略。当然不推荐就是了。ignore this strategy. It is not required to call this and this is only sugar.
  • success! – 策略成功。因为warden的策略有一个成功即会停止，所以紧接着会造成halt!也发动。
  • fail! – 策略失败。并发动halt！
  • redirect! – 跳转到另一个url。同样发动halt!
  • custom! – 返回一个自定义的rack array并造成halt!

感悟

当Devise太重的时候我们完全可以通过定制自己的warden strategy来定制我们自己的用户验证机制。
